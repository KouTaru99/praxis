17:Tf8c,# Đặc tả Yêu cầu — Mini-Wiki v1

> Phiên bản 1.0 (Chặng 1 — Ý tưởng & Yêu cầu)
> Phạm vi: v1 — ứng dụng wiki công khai, không xác thực người dùng

## 1. Yêu cầu Chức năng (Functional Requirements)

### 1.1 Quản lý Bài viết (Article)

| Mã | Yêu cầu |
|---|---|
| FR-1 | Xem danh sách bài viết đã đăng (tiêu đề, slug, ngày đăng), sắp xếp mới nhất trước. |
| FR-2 | Xem chi tiết bài viết qua URL /articles/:slug; Markdown render thành HTML an toàn. |
| FR-3 | Tạo bài viết mới: title, content (Markdown), published_at (tùy chọn). slug tự sinh từ title, hỗ trợ tiếng Việt. |
| FR-4 | Chỉnh sửa tiêu đề, nội dung, ngày đăng của bài viết. |
| FR-5 | Xóa bài viết; liên kết article_tags xóa theo (cascade). |
| FR-6 | Ngăn hai bài cùng slug; trả 409 Conflict khi trùng. |

### 1.2 Quản lý Nhãn (Tag)

| Mã | Yêu cầu |
|---|---|
| FR-7 | Xem toàn bộ danh sách tag. |
| FR-8 | Tạo tag mới (name); slug tự sinh, phải duy nhất. |
| FR-9 | Xóa tag. Nếu tag còn gắn bài viết thì từ chối, trả 409 (ON DELETE RESTRICT). |
| FR-10 | Gắn / gỡ tag khỏi bài viết. |
| FR-11 | Lọc danh sách bài viết theo một tag. |

### 1.3 Tìm kiếm (Search)

| Mã | Yêu cầu |
|---|---|
| FR-12 | Tìm bài theo từ khóa trong title lẫn content bằng Full-Text Search (FTS — tìm toàn văn). |
| FR-13 | Kết quả có phân trang (mặc định tối đa 20 kết quả/trang). |

### 1.4 Render & An toàn Nội dung

| Mã | Yêu cầu |
|---|---|
| FR-14 | Markdown parse bằng marked + làm sạch bằng DOMPurify trước khi trả HTML, ngăn XSS. |
| FR-15 | slug tiếng Việt chuẩn hóa đúng ("Học lập trình" thành hoc-lap-trinh) bằng slugify locale vi. |

## 2. Yêu cầu Phi Chức năng (Non-Functional Requirements)

| Mã | Yêu cầu |
|---|---|
| NFR-1 | Hiệu năng: CRUD đơn lẻ dưới 300ms local; FTS dưới 500ms nhờ GIN index. |
| NFR-2 | Bảo mật XSS: sanitize mọi Markdown; chặn script, event handler, iframe lạ. |
| NFR-3 | Bảo mật SQL Injection: mọi truy vấn qua Drizzle ORM (parameterized query). |
| NFR-4 | CORS: API chỉ chấp nhận origin của front-end (qua CORS_ORIGIN). |
| NFR-5 | Không xác thực (v1): không đăng ký/đăng nhập; nội dung công khai. |
| NFR-6 | Khả năng triển khai: container hóa bằng Docker; dev = prod (parity). |
| NFR-7 | Bảo trì: TypeScript strict; Drizzle migration version hóa; không lạm dụng any. |
| NFR-8 | Quan sát: mọi lỗi 5xx log ra stderr đủ context; không nuốt lỗi thầm. |

## 3. User Stories

| Mã | Story |
|---|---|
| US-1 | Là độc giả, tôi muốn xem danh sách bài mới nhất để biết có gì đáng đọc. |

| US-4 | Là tác giả, tôi muốn sửa bài đã đăng để cập nhật thông tin. |
| US-5 | Là tác giả, tôi muốn xóa bài không phù hợp để giữ wiki gọn. |
| US-6 | Là tác giả, tôi muốn gắn tag vào bài để phân loại theo chủ đề. |
| US-7 | Là độc giả, tôi muốn lọc bài theo tag để đọc đúng chủ đề quan tâm. |
| US-8 | Là độc giả, tôi muốn tìm bài theo từ khóa để tìm nhanh nội dung. |

18:Ta8d,# Sơ đồ ERD — Mini-Wiki v1

> ERD (Entity-Relationship Diagram): sơ đồ thực thể-quan hệ — mô tả có bảng nào, cột gì, nối nhau ra sao.

## 1. articles — Bài viết

| Cột | Kiểu | Ràng buộc | Ghi chú |
|---|---|---|---|
| id | INT | PK, AUTO_INCREMENT | Khóa chính nội bộ |
| title | VARCHAR(255) | NOT NULL | Tiêu đề bài viết |
| slug | VARCHAR(255) | NOT NULL, UNIQUE | Định danh trong URL, tự sinh từ title |
| content | TEXT | NOT NULL | Nội dung Markdown thuần |
| published_at | TIMESTAMPTZ | NULLABLE | NULL = bản nháp; có giá trị = đã đăng |
| created_at | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | Tự ghi khi tạo |
| updated_at | TIMESTAMPTZ | NOT NULL | Tự cập nhật khi sửa (Drizzle $onUpdate) |

Vì sao slug UNIQUE: slug dùng trong URL (/articles/:slug). Hai bài cùng slug gây xung đột routing, phải chặn ngay ở tầng DB.

Vì sao tách content (TEXT) khỏi title (VARCHAR): Markdown có thể dài hàng chục nghìn ký tự; danh sách bài chỉ cần title+slug, không kéo cả content giúp query nhẹ.

## 2. tags — Nhãn

| Cột | Kiểu | Ràng buộc | Ghi chú |
|---|---|---|---|
| id | INT | PK, AUTO_INCREMENT | Khóa chính |
| name | VARCHAR(100) | NOT NULL, UNIQUE | Tên hiển thị (ví dụ "JavaScript") |
| slug | VARCHAR(100) | NOT NULL, UNIQUE | Dùng trong URL filter (ví dụ javascript) |
| created_at | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | Tự ghi khi tạo |

## 3. article_tags — Bảng Nối (Junction Table)

Bảng trung gian thể hiện quan hệ nhiều-nhiều. Mỗi hàng = một cặp (bài viết, tag).

| Cột | Kiểu | Ràng buộc | Ghi chú |
|---|---|---|---|
| article_id | INT | FK to articles.id, ON DELETE CASCADE | Xóa bài thì liên kết tự xóa theo |
| tag_id | INT | FK to tags.id, ON DELETE RESTRICT | Tag còn liên kết bài thì KHÔNG cho xóa |
| (PK) | (article_id, tag_id) | PRIMARY KEY composite | Ngăn gắn trùng tag vào cùng một bài |

ON DELETE CASCADE vs RESTRICT:
- article_id CASCADE: xóa bài thì liên kết tag xóa theo (hợp lý).
- tag_id RESTRICT: xóa tag khi còn bài gắn thì DB từ chối, tầng app bắt lỗi và trả 409 Conflict.

## 4. Index cần tạo

| Index | Bảng | Cột | Loại | Mục đích |
|---|---|---|---|---|
| articles_slug_idx | articles | slug | UNIQUE B-Tree | Tìm nhanh bài theo slug |
| articles_fts_idx | articles | to_tsvector(title, content) | GIN | Full-Text Search |
| tags_slug_idx | tags | slug | UNIQUE B-Tree | Tìm tag theo slug |
| article_tags_tag_idx | article_tags | tag_id | B-Tree | Lọc bài theo tag hiệu quả |
19:Tc5b,# Hợp đồng API (API Contract) — Mini-Wiki v1

> API Contract: tài liệu mô tả giao kèo giữa Front-end và Back-end. Đây là bản phác thảo, sẽ hình thức hóa thành OpenAPI 3.0 ở Chặng 2.
> Base URL: http://localhost:3000/api (dev)
> Định dạng: JSON (Content-Type: application/json)

## 1. Articles — Bài viết

| # | Method | Endpoint | Mô tả |
|---|---|---|---|
| A1 | GET | /articles | Lấy danh sách bài viết (hỗ trợ ?tag=, ?page=, ?limit=) |
| A2 | POST | /articles | Tạo bài viết mới |
| A3 | GET | /articles/:slug | Lấy chi tiết một bài viết theo slug |
| A4 | PUT | /articles/:slug | Cập nhật bài viết |
| A5 | DELETE | /articles/:slug | Xóa bài viết |

Ví dụ response GET /articles (lưu ý: KHÔNG trả content để giảm payload):

    {
      "data": [
        { "id": 1, "title": "Học Git cơ bản", "slug": "hoc-git-co-ban",
          "published_at": "2026-06-28T07:00:00.000Z",
          "tags": [ { "id": 2, "name": "Git", "slug": "git" } ] }
      ],
      "meta": { "total": 42, "page": 1, "limit": 20 }
    }

Ví dụ body POST /articles:

    {
      "title": "Học Git cơ bản",
      "content": "## Giới thiệu\nGit là hệ thống quản lý phiên bản...",
      "published_at": "2026-06-28T07:00:00.000Z",
      "tagIds": [2, 5]
    }

- title (string, bắt buộc, tối đa 255), content (string, bắt buộc, Markdown)
- published_at (ISO 8601, tùy chọn, NULL = nháp), tagIds (int[], tùy chọn)
- Response 201: object Article đầy đủ. Lỗi: 400 (thiếu/sai trường), 409 (slug trùng), 500.

GET /articles/:slug trả Article đầy đủ (có content). Lỗi: 404, 500.
PUT /articles/:slug: mọi trường tùy chọn; nếu gửi tagIds thì THAY THẾ toàn bộ tag (không merge). Lỗi: 400, 404, 409, 500.
DELETE /articles/:slug: trả 204 No Content; article_tags xóa cascade. Lỗi: 404, 500.

## 2. Tags — Nhãn

| # | Method | Endpoint | Mô tả |
|---|---|---|---|
| T1 | GET | /tags | Lấy danh sách toàn bộ tag |
| T2 | POST | /tags | Tạo tag mới (body: { "name": "TypeScript" }) |
| T3 | DELETE | /tags/:id | Xóa tag theo ID |

- T2 Response 201: { id, name, slug, created_at }. Lỗi: 400, 409 (tên/slug trùng), 500.
- T3 Response 204. Lỗi: 404, 409 (tag còn liên kết bài viết — TAG_IN_USE), 500.

## 3. Search — Tìm kiếm

| # | Method | Endpoint | Mô tả |
|---|---|---|---|
| S1 | GET | /search?q=tu-khoa | Full-Text Search trên title + content |

- q (string, bắt buộc), page/limit (tùy chọn). Lỗi: 400 (thiếu q), 500.

## 4. Bảng mã lỗi

Mọi lỗi theo cấu trúc: { "error": { "code": "SLUG_CONFLICT", "message": "..." } }

| HTTP | Khi nào | Ví dụ code |
|---|---|---|
| 400 Bad Request | Thiếu trường, sai kiểu, vi phạm validation | MISSING_FIELD, QUERY_REQUIRED |
| 404 Not Found | Không tìm thấy article/tag | ARTICLE_NOT_FOUND, TAG_NOT_FOUND |
| 409 Conflict | Slug/tên trùng; hoặc xóa tag còn liên kết | SLUG_CONFLICT, TAG_IN_USE |
| 500 Internal Server Error | Lỗi server không mong muốn | INTERNAL_ERROR |

## Bối cảnh (Context)
Mini-Wiki là ứng dụng wiki công khai CRUD phục vụ học tập. Cần chọn runtime + framework Back-end và framework Front-end.
Ràng buộc: ưu tiên 1 ngôn ngữ xuyên suốt (giảm friction cho người học); tách FE/BE để dạy rõ API boundary; v1 không cần SSR.

## Các lựa chọn (Options)

Back-end:
| Phương án | Dẫn chứng |
|---|---|
| Node.js 20 + Express 5 + TypeScript (CHỌN) | Node phổ biến nhất (Stack Overflow 2024); Express ecosystem lớn nhất; TypeScript giảm bug runtime |
| Spring Boot (Java) | Barrier học cao (JVM, Gradle, verbose); hợp enterprise nhưng quá nặng cho bài học |
| Go + Fiber | Hiệu năng tốt nhưng ecosystem nhỏ, phải học thêm Go |
| Python + FastAPI | Async tốt nhưng dynamic typing mặc định |

Front-end:
| Phương án | Dẫn chứng |
|---|---|
| React 18 + Vite 6 (CHỌN) | React #1 framework (SO 2024); Vite HMR ~50ms, nhanh gấp 10-20 lần webpack |
| Next.js 14 | Tốt cho production nhưng làm mờ ranh giới FE/BE, khó dạy API contract |
| Vue 3 + Vite | Dễ học nhưng thị phần & tài liệu nhỏ hơn React |

## Quyết định (Decision)
Node.js 20 + Express 5 + TypeScript (BE) và React 18 + Vite 6 (FE). Lý do cốt lõi: 1 ngôn ngữ TypeScript cho cả 2 đầu; Express tối giản nên người học thấy rõ vòng đời request/response; tách api/ và web/ làm API boundary hiện rõ.

## Hệ quả (Consequences)
Tích cực: cùng kiến thức TypeScript cho BE+FE; async/await native; npm ecosystem khổng lồ.
Đánh đổi: Node single-thread (không hợp tác vụ nặng CPU); Express ít opinionated (tự tổ chức folder); React SPA yếu SEO (chấp nhận, v1 không ưu tiên SEO).

// Lỗi PostgreSQL (qua driver `pg`) có SQLSTATE `code` + tên `constraint` vi phạm.
interface PgError {
  code: string;
  constraint?: string;
}

function hasStringCode(err: unknown): err is PgError {
  return typeof err === 'object' && err !== null && 'code' in err
    && typeof (err as { code: unknown }).code === 'string';
}

// Drizzle bọc lỗi driver gốc vào DrizzleQueryError, lỗi pg thật nằm ở
// `.cause` (chuẩn Error cause-chaining của Node) — bóc dần tới khi gặp.
function extractPgError(err: unknown): PgError | undefined {
  if (hasStringCode(err)) return err;
  if (typeof err === 'object' && err !== null && 'cause' in err) {
    return extractPgError((err as { cause: unknown }).cause);
  }
  return undefined;
}

// Error middleware của Express PHẢI có đủ 4 tham số (err, req, res, next)
export function errorHandler(
  err: unknown,
  _req: Request,
  res: Response,
  _next: NextFunction,
): void {
  if (err instanceof AppError) {
    res.status(err.statusCode).json({ error: { code: err.code, message: err.message } });
    return;
  }

  const pgError = extractPgError(err);
  if (pgError) {
    if (pgError.code === '23505') {
      const code = pgError.constraint?.includes('slug') ? 'SLUG_CONFLICT' : 'TAG_NAME_CONFLICT';
      res.status(409).json({ error: { code, message: `Giá trị đã tồn tại (constraint: ${pgError.constraint}).` } });
      return;
    }
    if (pgError.code === '23503') {
      res.status(409).json({ error: { code: 'TAG_IN_USE', message: 'Tag đang gắn bài viết, không thể xóa.' } });
      return;
    }
  }

  console.error('[ERROR 500]', err); // NFR-8: không nuốt lỗi thầm
  res.status(500).json({ error: { code: 'INTERNAL_ERROR', message: 'Đã xảy ra lỗi phía server.' } });
}

async function apiSend<T>(method: string, path: string, body?: unknown): Promise<T> {
  const res = await fetch(`${BASE_URL}${path}`, {
    method,
    headers: { 'Content-Type': 'application/json' },
    body: body !== undefined ? JSON.stringify(body) : undefined,
  });
  if (res.status === 204) return undefined as T; // DELETE: không có body
  if (!res.ok) {
    const b = (await res.json().catch(() => null)) as { error?: { message?: string } } | null;
    throw new Error(b?.error?.message ?? `HTTP ${res.status}`);
  }
  return res.json() as Promise<T>;
}

// AppError: lỗi có chủ ý từ business logic (controller ném ra thay vì lỗi DB thô)
export class AppError extends Error {
  constructor(
    public readonly statusCode: number,
    public readonly code: string,
    message: string,
  ) {
    super(message);
    this.name = 'AppError';
  }
}

// Lỗi PostgreSQL (qua driver `pg`) có SQLSTATE `code` + tên `constraint` vi phạm.
interface PgError {
  code: string;
  constraint?: string;
}

function hasStringCode(err: unknown): err is PgError {
  return typeof err === 'object' && err !== null && 'code' in err
    && typeof (err as { code: unknown }).code === 'string';
}

// Drizzle bọc lỗi driver gốc vào DrizzleQueryError, lỗi pg thật nằm ở
// `.cause` (chuẩn Error cause-chaining của Node) — bóc dần tới khi gặp.
function extractPgError(err: unknown): PgError | undefined {
  if (hasStringCode(err)) return err;
  if (typeof err === 'object' && err !== null && 'cause' in err) {
    return extractPgError((err as { cause: unknown }).cause);
  }
  return undefined;
}

// Error middleware của Express PHẢI có đủ 4 tham số (err, req, res, next)
export function errorHandler(
  err: unknown,
  _req: Request,
  res: Response,
  _next: NextFunction,
): void {
  if (err instanceof AppError) {
    res.status(err.statusCode).json({ error: { code: err.code, message: err.message } });
    return;
  }

  const pgError = extractPgError(err);
  if (pgError) {
    if (pgError.code === '23505') {
      const code = pgError.constraint?.includes('slug') ? 'SLUG_CONFLICT' : 'TAG_NAME_CONFLICT';
      res.status(409).json({ error: { code, message: `Giá trị đã tồn tại (constraint: ${pgError.constraint}).` } });
      return;
    }
    if (pgError.code === '23503') {
      res.status(409).json({ error: { code: 'TAG_IN_USE', message: 'Tag đang gắn bài viết, không thể xóa.' } });
      return;
    }
  }

  console.error('[ERROR 500]', err); // NFR-8: không nuốt lỗi thầm
  res.status(500).json({ error: { code: 'INTERNAL_ERROR', message: 'Đã xảy ra lỗi phía server.' } });
}
