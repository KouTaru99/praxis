Chặng 2 · Thiết kế & ADR

Hình thức hóa API thành OpenAPI 3.0

Mục tiêu

Biến bản phác API contract (Chặng 1) thành file OpenAPI chuẩn, máy đọc được.

Sản phẩm của bước: docs/openapi.yaml mô tả 9 endpoint theo chuẩn OpenAPI 3.0.3.

Sau bước này bạn sẽ thấy: Bạn sẽ có một hợp đồng API chính thức — nền tảng để sinh tài liệu tương tác, validate request, hay generate client tự động.

Hướng dẫn

Ở Chặng 1 ta phác docs/api.md bằng văn xuôi cho người đọc. Giờ ta nâng nó thành OpenAPI (tên cũ: Swagger) — chuẩn mô tả REST API dạng máy đọc được (YAML). Vì sao đáng công sức này? Một file OpenAPI cho phép: sinh tài liệu tương tác (Swagger UI để bấm thử API ngay trên trình duyệt), validate request/response tự động, và generate client SDK cho front-end. Hợp đồng giờ không chỉ là chữ — nó là spec thực thi được.

Tạo docs/openapi.yaml. Phần khai báo đầu + một endpoint mẫu (GET /articles):

docs/openapi.yaml
openapi: "3.0.3"

info:
  title: Mini-Wiki API
  version: "1.0.0"
  description: |
    REST API công khai cho Mini-Wiki — CRUD bài viết, nhãn và tìm kiếm toàn văn.
    Không có xác thực người dùng ở v1.

servers:
  - url: http://localhost:3000/api
    description: Dev local

tags:
  - name: articles
    description: Quản lý bài viết
  - name: tags
    description: Quản lý nhãn
  - name: search
    description: Tìm kiếm toàn văn

paths:
  /articles:
    get:
      tags: [articles]
      summary: Lấy danh sách bài viết
      parameters:
        - name: tag
          in: query
          description: Slug của tag để lọc
          required: false
          schema: { type: string }
        - name: page
          in: query
          schema: { type: integer, default: 1, minimum: 1 }
        - name: limit
          in: query
          schema: { type: integer, default: 20, minimum: 1, maximum: 100 }
      responses:
        "200":
          description: Danh sách bài viết (không gồm content để giảm payload)
    # ... POST /articles, /articles/{slug} (GET/PUT/DELETE),
    # ... /tags (GET/POST), /tags/{id} (DELETE), /search (GET)
    # Xem đầy đủ 9 endpoint ở commit đối chiếu bên dưới.

File đầy đủ định nghĩa cả 9 endpoint (5 Articles + 3 Tags + 1 Search) cùng các components/schemas (Article, Tag, Error) để tái dùng. Mỗi endpoint khai báo rõ tham số, body, và mọi mã lỗi có thể xảy ra — đúng như bảng mã lỗi đã phác ở Chặng 1.

Mẹo: Mẹo nghề: giữ file OpenAPI đồng bộ với code là thử thách thật. Nhiều team dùng cách ngược lại — sinh OpenAPI từ code (qua decorator/annotation) để spec không bao giờ lỗi thời. Ở quy mô bài học này, viết tay trước giúp bạn hiểu rõ từng phần của một spec; ở dự án lớn, cân nhắc tự động hóa.
