17:T20b4,# Market Pulse

Nền tảng cảnh báo thị trường real-time theo mô hình B2B. Tenant đăng ký rule (ngưỡng giá, thay đổi %, đột biến volume), khi điều kiện khớp hệ tự động gọi webhook. Dữ liệu đến từ Binance WebSocket (~80 trades/giây với BTC/USDT).

**Mục đích:** Đây là repo mẫu cho Lab lớn trên nền tảng Praxis — người học xây từng bước theo 10 chặng, mỗi chặng kết thúc bằng milestone có thể chạy thật.

## Compose profile

Mỗi profile nhóm các service liên quan — khởi động từng phần mà không cần bật toàn bộ JVM từ Chặng 1.

| Profile | Service khởi động | Mục đích |
|---------|-------------------|---------|
| `core` | gateway, identity, identity-db | Auth + routing — Chặng 2 |
| `feed` | market-feed, rabbitmq | Dòng dữ liệu Binance — Chặng 3 |
| `processing` | aggregation, aggregation-db, rule-engine, rule-engine-db, rabbitmq | Aggregation + rule eval — Chặng 4–5 |
| `delivery` | alert-delivery, alert-delivery-db, billing, billing-db, mailhog, rabbitmq | Delivery + billing — Chặng 6–7 |
| `full` | tất cả + dashboard | End-to-end — Chặng 8–10 |

> **Lưu ý:** Các profile trên sẽ hoàn thiện dần theo chặng. Ở Chặng 1, file `docker-compose.yml` chưa tồn tại — được build từ Chặng 2 trở đi.

```bash
# Ví dụ chạy profile core (Chặng 2 trở đi)
docker compose --profile core up -d

# Kiểm tra health
docker compose ps

# Xem log service cụ thể
docker compose logs -f gateway

# Dừng
docker compose --profile core down
```

## 1. Vấn đề cần giải quyết

Doanh nghiệp cần theo dõi biến động giá/volume trên thị trường tiền mã hoá. Giải pháp hiện tại là polling định kỳ hoặc xây custom script — tốn chi phí vận hành, dễ sót sự kiện, không scale.

**Market Pulse** cung cấp hạ tầng cảnh báo real-time theo mô hình B2B:
- Tenant đăng ký rule theo symbol + điều kiện.
- Khi điều kiện khớp với dữ liệu live (~80 trades/giây từ Binance), hệ tự động gửi webhook.
- Tenant không cần viết code polling, không cần quản lý kết nối WS, không cần lo retry khi delivery thất bại.

**Chi phí nếu không có:** tự xây = tốn kỹ sư, khó maintain, miss nhiều edge case (reconnect WS, retry webhook, quota enforcement).

## 2. Goals

| # | Outcome | Đo bằng |
|---|---------|---------|
| G1 | Hệ nhận dữ liệu Binance thật và xử lý real-time | `docker compose --profile full up` → 7 JVM + DB + RabbitMQ healthy; log trade Binance đầu tiên trong 10 giây |
| G2 | Tenant tự quản lý rule và theo dõi alert không cần gọi API thủ công | Luồng tạo rule → alert trigger → webhook DELIVERED hoàn chỉnh trong Dashboard |
| G3 | Hệ vận hành đúng quota — vượt hạn mức thì chặn rõ ràng, không drop silently | Rule thứ (limit+1) → HTTP 402; alert quota vượt → eval pause, không mất alert |
| G4 | Webhook delivery tin cậy — fail tự retry, có audit trail đầy đủ | Retry 3 lần (1s → 4s → 16s); history đủ attempted_at + status_code + status |
| G5 | Hệ phân tách module rõ ràng, mỗi service độc lập DB riêng | Không service nào query DB của service khác; giao tiếp chỉ qua REST internal + RabbitMQ |

## 3. Non-Goals

| Điều không làm | Lý do |
|---------------|-------|
| Thanh toán thật (cổng payment) | Billing chỉ cần đếm usage + quota + xuất thông tin; không tích hợp PSP |
| Multi-region / HA / Kubernetes | Nằm ngoài phạm vi nhập môn; ghi nhận để phát triển sau |
| SMS delivery | Webhook + email giả lập đủ minh hoạ delivery pipeline; SMS tăng chi phí API |
| Rule phức tạp (AND/OR, RSI/MACD, chỉ báo kỹ thuật) | v1 chỉ cần 3 loại rule; rule phức tạp là lab kế thừa |
| Nguồn dữ liệu bổ sung (BrandWatch, Wikipedia) | Adapter tách bạch trong `market-feed` để sau thay Binance không đập lõi — nhưng implementation để sau |

## 4. User Stories (Persona B — Tenant B2B)

### Rule management

- **B1 (P0):** Là tenant B2B, muốn tạo alert rule (price threshold / % change / volume spike) cho symbol quan tâm, để hệ thống tự theo dõi thay vì phải polling thủ công.
- **B2 (P0):** Là tenant B2B, muốn nhận webhook call với alert payload khi rule trigger, để tự động hoá hành động downstream không cần polling.
- **B3 (P0):** Là tenant B2B, muốn xem alert history và delivery status (delivered / failed / retrying), để audit độ tin cậy của hệ.

### Usage & quota

- **B4 (P1):** Là tenant B2B, muốn xem usage hiện tại (active rules, alerts fired this month) so với quota, để lên kế hoạch capacity.

### Edge cases

- **B5 (P0):** Là tenant B2B, khi webhook delivery thất bại, muốn hệ tự retry với exponential backoff, để failure tạm thời không đồng nghĩa mất alert.
- **B6 (P0):** Là tenant B2B, nếu vượt quota, muốn rule evaluation pause (không drop silently), để có cơ hội nâng gói hoặc giảm rule thủ công.

## 5. Requirements

### P0 — Must-have

---

#### R01 — Hạ tầng repo & Docker Compose

**Mô tả:** Mono-repo Gradle multi-project; 7 service Spring Boot + FE React/Vite + `infra/`.

**Docker Compose profile:**

| Profile | Service |
|---------|---------|
| `core` | gateway, identity, identity-db |
| `feed` | market-feed, rabbitmq |
| `processing` | aggregation, aggregation-db, rule-engine, rule-engine-db, rabbitmq |
| `delivery` | alert-delivery, alert-delivery-db, billing, billing-db, mailhog, rabbitmq |
| `full` | tất cả + dashboard |

**Acceptance Criteria:**
- `docker compose --profile core up -d` → gateway :8080 + identity :8081 healthy.
- Mỗi profile `up` không lỗi `cannot connect to <service ngoài profile>`.
- Service nào thiếu dependency ngoài profile → log WARN và tiếp tục, không crash JVM.

---

#### R02 — `market-feed`: 3 chế độ hoạt động

**Mô tả:** Connector Binance WebSocket với 3 mode runtime.

| Mode | Hành vi |
|------|---------|
| `live` | Kết nối Binance WS public; chuẩn hoá trade → TradeEvent; publish queue; auto-reconnect khi mất kết nối |
| `record` | Giống `live` + ghi NDJSON ra file với timestamp gốc |
| `replay` | Đọc file NDJSON đã record; phát lại với multiplier ×N; giữ tỉ lệ thời gian tương đối |

**Acceptance Criteria:**
- Chuyển mode bằng env `FEED_MODE=live|record|replay`.
- `replay` ×3: phát đúng thứ tự timestamp, hoàn thành trong 1/3 thời gian gốc (±10%).
- `live` mất kết nối WS → log WARN, không crash JVM, tự reconnect trong 30 giây.

---

#### R03 — `rule-engine`: 3 loại rule

**Mô tả:** Evaluate rule theo trade stream và candle stream real-time.

| Loại rule | Điều kiện trigger |
|-----------|-------------------|
| `PRICE_THRESHOLD` | Symbol X vượt/tụt mức giá P (so với trade mới nhất) |
| `PERCENT_CHANGE` | Symbol X thay đổi ≥ p% trong N phút (dùng dữ liệu aggregation) |
| `VOLUME_SPIKE` | Volume 1 phút ≥ k× trung bình 60 phút (dùng dữ liệu aggregation) |

**Acceptance Criteria:**
- `PRICE_THRESHOLD` kích hoạt trong vòng 1 giây kể từ trade vượt ngưỡng.
- `PERCENT_CHANGE` chính xác ±0.1% so với tính tay trên cùng window.
- `VOLUME_SPIKE` không false-positive khi volume đều đặn.
- Rule bị EVAL_PAUSED (do quota) → không trigger, không mất dữ liệu.

---

#### R04 — `alert-delivery`: webhook + retry

**Mô tả:** Gửi HTTP POST đến endpoint webhook của tenant khi rule trigger; xử lý retry và audit.

**Payload webhook:**
```json
{
  "alertId": "uuid",
  "ruleId": "uuid",
  "symbol": "BTCUSDT",
  "ruleType": "PRICE_THRESHOLD",
  "triggeredAt": "2026-07-04T10:00:00Z",
  "triggerValue": "49850.00"
}
```

**Retry logic:**
- `2xx` → status `DELIVERED`
- `4xx` → status `CLIENT_ERROR`, không retry
- `5xx` / timeout → retry sau 1s, 4s, 16s; sau 3 lần → `FAILED`

**Acceptance Criteria:**
- Alert history lưu đủ: attempted_at, http_status, status mỗi attempt.
- Sau 3 lần thất bại: status = FAILED, không retry thêm.
- Alert vẫn xuất hiện trong history kể cả khi FAILED.

---

#### R05 — `identity` + `gateway`: Tenant và API key

**Mô tả:** Tenant quản lý API key; gateway xác thực mọi request trước khi forward.

**Auth flow:**
1. Gateway nhận `X-API-Key: <raw-key>`.
2. Tính SHA-256 hash; tra cache Caffeine (TTL 5s).
3. Cache miss → gọi `identity GET /internal/keys/{hash}`.
4. Key hợp lệ → thêm `X-Tenant-Id`, xoá `X-API-Key`, forward.
5. Key không tồn tại / revoked → 401.

**Acceptance Criteria:**
- Request không có key → 401.
- Key sai → 401.
- Key bị revoke → 401 trong vòng 5 giây (TTL cache hết).
- Request hợp lệ → forward giữ nguyên body.

---

#### R06 — `billing`: usage + quota

**Mô tả:** Đếm usage theo tenant; enforce quota theo gói.

**Quota enforcement:**
- **Rule quota (sync):** rule-engine check `billing GET /internal/quota/{tenantId}` khi tạo rule; cache 30s.
- **Alert quota (async event-driven):** billing consume AlertTriggeredEvent → đếm; khi vượt → phát `QuotaExceededEvent` → rule-engine pause eval tenant.

**Acceptance Criteria:**
- `usage.alertsFired` khớp đếm tay từ alert_history.
- Vượt quota rule → HTTP 402 khi tạo rule thứ (limit+1); rule không được tạo.
- Usage reset đúng đầu kỳ (tháng).
- Vượt quota alert → rule eval pause, không drop alert đã trigger trước đó.

---

#### R07 — `dashboard` (FE portal)

**Mô tả:** React + Vite SPA — giao diện quản lý đầy đủ cho tenant.

**Tính năng:**
- Quản lý API key: tạo, xem danh sách, thu hồi.
- Rule CRUD: tạo (symbol + type + params + webhookUrl), xem danh sách, xoá.
- Alert history: phân trang, filter; xem delivery attempts từng alert.
- Usage widget: active rules vs limit, alerts fired vs limit (progress bar).

**Acceptance Criteria:**
- Luồng "tạo rule → thấy alert trigger → xem webhook DELIVERED" hoàn chỉnh trong UI, không cần gọi API thủ công.
- Dashboard hiển thị thông báo rõ khi vượt quota.
- API key raw chỉ hiển thị 1 lần ngay sau khi tạo.

### Happy path — luồng end-to-end

```

  And: tenant T1 đã tạo, API key K1 issued
  And: rule R1 = PRICE_THRESHOLD BTC/USDT BELOW 50000
        webhook = http://localhost:9999/hook

Then:  trong vòng 2 giây, alert-delivery gửi POST đến localhost:9999/hook
  And: payload chứa ruleId=R1, symbol=BTCUSDT, triggeredAt (ISO-8601), triggerValue < 50000
  And: alert history trong dashboard hiển thị status=DELIVERED
  And: billing tăng alertsFired cho tenant T1 thêm 1
```

### Retry khi webhook fail

```
Given: endpoint webhook trả HTTP 500 liên tục
When:  rule trigger → alert-delivery nhận AlertTriggeredEvent
Then:  thử lại sau 1s, thử lại sau 4s, thử lại sau 16s
  And: sau lần thứ 3 thất bại → status = FAILED, không retry thêm
  And: alert xuất hiện trong history với status = FAILED và attemptCount = 3
```

### Quota rule vượt mức

```
Given: tenant T1 có gói giới hạn 2 active rules
  And: T1 đã tạo 2 rule đang ACTIVE
When:  T1 gọi POST /api/v1/rules để tạo rule thứ 3
Then:  gateway/rule-engine trả HTTP 402 { "error": "QUOTA_EXCEEDED_RULES" }
  And: không có rule nào được tạo trong DB
  And: dashboard hiển thị thông báo vượt quota
```

### Compose profile độc lập

```
Given: docker compose --profile feed up -d
When:  market-feed khởi động
Then:  market-feed + rabbitmq healthy
  And: không lỗi "cannot connect to gateway/identity/aggregation"
  And: market-feed log được ít nhất 1 trade từ Binance WS trong 10 giây
```

### Replay mode

```
Given: file record có 1000 trade BTC/USDT ghi ở tốc độ thật (T giây)
  And: FEED_MODE=replay, REPLAY_MULTIPLIER=3
When:  market-feed khởi động chế độ replay
Then:  1000 trade phát hết trong khoảng T/3 giây (±10%)

```

## Context

Hệ Market Pulse cần một message broker (phần mềm trung gian nhận và phân phối message giữa các service) để:

1. **Fanout trade stream** — 1 nguồn (`market-feed`) → nhiều consumer (`aggregation`, `rule-engine`).
2. **Async delivery pipeline** — `rule-engine` → `alert-delivery` + `billing` mà không gọi trực tiếp.
3. **Retry / dead-letter** — `alert-delivery` cần cơ chế xử lý message thất bại.

**Ràng buộc cứng:** Lab chạy trên laptop cá nhân (16 GB RAM) cùng 7 JVM + 5 PostgreSQL + broker. Tổng RAM phải ≤ ~4.5 GB để thoải mái — không bao gồm OS + browser.

---

## Các lựa chọn

### Phương án A — Apache Kafka 3.7 (KRaft mode)

**Kafka** là distributed streaming platform (nền tảng stream phân tán) — dữ liệu lưu dạng log, consumer đọc theo offset, có thể replay từ bất kỳ điểm nào.

| Tiêu chí | Đánh giá |
|----------|----------|
| Adoption | ~500M Docker Hub pulls (confluent/kafka) |
| RAM broker | ~600–900 MB (KRaft mode, không ZooKeeper) |
| RAM tổng (7 JVM + 5 PG + broker) | ~4.5–5.0 GB — sát hoặc vượt ngưỡng |
| Retry / Dead-letter queue (DLQ) | Không native — tự implement bằng topic riêng |
| Topic/partition setup | Trung bình: tạo topic + partition trước khi chạy |
| Replay dữ liệu thị trường | Built-in (consumer offset reset) |
| Spring support | `spring-kafka` 9 năm; production-proven |
| Learning value | Cao nhất cho distributed systems |

**Trade-off:** RAM quá lớn cho lab laptop; DLQ phải tự implement tăng độ phức tạp.

---

### Phương án B — RabbitMQ 3.13

**RabbitMQ** là message broker truyền thống (broker trung gian nhận/gửi message) — model push-based, hỗ trợ nhiều loại exchange (fanout, topic, direct, headers).

| Tiêu chí | Đánh giá |
|----------|----------|
| Adoption | Docker Hub 1B+ pulls |
| RAM broker | ~150–200 MB |
| RAM tổng (7 JVM + 5 PG + broker) | ~3.7–4.2 GB ✓ |
| Retry / Dead-letter queue (DLQ) | **Native** — `x-dead-letter-exchange`, `x-message-ttl` |
| Setup | Thấp: 1 container, Management UI :15672 có sẵn |
| Replay built-in | Không — nhưng `market-feed replay mode` thay thế |
| Spring support | `spring-boot-starter-amqp` 14 năm; battle-tested |
| Learning value | Tốt cho pattern queue/exchange; thiếu offset/partition |

---

### So sánh tổng hợp

| Tiêu chí | RabbitMQ | Kafka |
|----------|----------|-------|
| RAM broker | **~175 MB** | ~750 MB |
| RAM tiết kiệm | — | tốn thêm ~575 MB |
| Phù hợp RAM 16 GB laptop | ✓ Thoải mái | ⚠ Sát ngưỡng |
| DLQ / retry native | ✓ | ✗ |
| Replay dữ liệu thị trường | Qua `market-feed` | Built-in |
| Độ phức tạp setup | Thấp | Trung bình |
| Spring maturity | 14 năm | 9 năm |

---

## Quyết định

**Chọn RabbitMQ 3.13.**

Lý do quyết định:

1. **RAM là ràng buộc cứng** — tiết kiệm ~500–575 MB so với Kafka; profile `full` vào khoảng 3.7–4.2 GB thay vì 4.5–5.0 GB. Với laptop 16 GB chạy OS + browser + Docker, khoảng cách này quan trọng.

2. **DLQ native** — pattern retry/dead-letter của `alert-delivery` implement trực tiếp bằng `x-dead-letter-exchange`, không phải tự viết logic.

3. **Replay dữ liệu** — `market-feed replay mode` (đọc NDJSON, phát lại theo multiplier) đáp ứng hoàn toàn nhu cầu lab mà không cần broker replay.

4. **Tất cả pattern cần dạy đều implement được:** async messaging, fanout, topic routing, multiple consumer, retry/DLQ.

---

## Consequences

**Tích cực:**
- Toàn bộ pattern dạy trong Lab đều cover: fanout exchange, topic routing, direct delivery, retry + DLQ.
- Management UI :15672 là công cụ quan sát trực quan cho Chặng 9 (queue depth, message rate).
- Replay dữ liệu thị trường giải quyết ở tầng `market-feed` — tách bạch hơn.

**Đánh đổi / hạn chế:**
- Không dạy được consumer group offset, partition, log retention — các concept cốt lõi của Kafka.
- Lab ghi nhận rõ: **"Khi nào cần Kafka"** — khi cần: (a) nhiều consumer group độc lập replay từ đầu, (b) retention dữ liệu dài hạn, (c) throughput > 100k msg/s. Market Pulse không đáp ứng điều kiện nào trong số này.
- Nếu sau này scale lên Kafka, phải thay toàn bộ `spring-amqp` bằng `spring-kafka` — không drop-in.

## Context

Market Pulse gồm 7 service Spring Boot + 1 FE + infra. Cần quyết định chứa tất cả trong 1 repo (mono-repo) hay tách ra từng repo riêng (multi-repo).

**Ràng buộc đặc thù của Lab:** Mỗi bước trong 10 chặng cần một `commit_ref` — con trỏ đến commit mẫu để người học diff khi bị kẹt. Đây là xương sống của Lab, ảnh hưởng trực tiếp đến lựa chọn cấu trúc repo.

---

## Các lựa chọn

### Phương án A — Multi-repo (8 repo riêng biệt)

Mỗi service là 1 repository GitHub độc lập. `dashboard/` và `infra/` thêm 2 repo nữa.

| Tiêu chí | Đánh giá |
|----------|----------|
| Clone | 8 lệnh `git clone` hoặc 1 repo tổng dùng submodule |
| Docker Compose | Phải bind-mount từ nhiều thư mục khác nhau |
| Commit-per-step | 1 bước Lab = cập nhật 1–3 repo → nhiều commit ID, khó đối chiếu |
| Ranh giới service | Physical enforce — không thể import chéo về mặt file system |
| CI/CD | Trigger riêng per repo; phức tạp khi thay đổi cross-service |
| Onboarding | Người học cần clone nhiều repo trước khi bắt đầu bước đầu tiên |

**Hạn chế chính:** `commit_ref` trải rải 8 repo — người học phải tra nhiều nơi khi stuck.

---

### Phương án B — Mono-repo, Gradle multi-project

Một repo duy nhất; mỗi service là Gradle subproject (`:gateway`, `:identity`, ...). `dashboard/` là thư mục Node riêng, không phải Gradle subproject.

| Tiêu chí | Đánh giá |
|----------|----------|
| Clone | 1 lệnh `git clone` |
| Docker Compose | `infra/docker-compose.yml` tham chiếu context tương đối — tự nhiên |
| Commit-per-step | **1 commit ID = toàn bộ trạng thái hệ** — người học `git checkout <hash>` là đủ |
| Ranh giới service | Enforce bằng quy ước Gradle: `common/` duy nhất được share; service khác không import chéo |
| CI/CD | 1 pipeline, filter path để build chỉ service thay đổi |
| Onboarding | 1 lệnh, 1 IDE window |

**Tham chiếu:** `spring-projects/spring-petclinic-microservices` (5.2k ⭐) dùng cùng pattern.

---

### So sánh tổng hợp

| Tiêu chí | Mono-repo | Multi-repo |
|----------|-----------|------------|
| commit_ref cho Lab | ✓ 1 hash = 1 state hoàn chỉnh | ✗ Trải rải N repo |
| Onboarding | ✓ 1 lệnh | ✗ N lệnh |
| Docker Compose | ✓ Tự nhiên | ✗ Phức tạp bind-mount |
| Ranh giới service | Convention (enforce được qua Gradle config) | Physical |
| Scale về sau | Có thể extract khi thật sự cần | Từ đầu đã tách |

---

## Quyết định

**Chọn mono-repo, Gradle multi-project.**

Lý do:

1. **`commit_ref` là xương sống Lab** — chỉ mono-repo cho phép 1 commit ID map đến trạng thái đầy đủ của tất cả service. Đây là tính năng không thể đánh đổi.

2. **Onboarding tối thiểu** — 1 `git clone`, mở 1 IDE window, chạy 1 `docker compose`.

3. **Ranh giới service** vẫn enforce được bằng quy tắc Gradle — mạnh hơn quy ước văn bản.

## Các lựa chọn

### Phương án A — Gateway check quota cho mọi request

Gateway hỏi `billing` mỗi khi có request tạo rule / mỗi khi alert trigger — trước khi forward.

| Tiêu chí | Đánh giá |
|----------|----------|
| Latency | Mỗi request thêm 1 REST call đến billing (thường ~5–10ms trong Docker network) |
| Coupling | Gateway phải biết về quota logic của từng business operation — vi phạm SRP |
| Consistency | Strong — quyết định luôn dựa trên data mới nhất |
| Scaling | Gateway trở thành bottleneck nếu billing chậm; phải cache hoặc timeout xử lý phức tạp |

**Hạn chế chính:** Gateway nên stateless và chỉ lo auth + routing. Nhúng quota logic vào Gateway làm phình trách nhiệm và tạo tight coupling với billing schema.

---

### Phương án B — Rule-engine check quota khi tạo rule (sync); Alert quota hoàn toàn sync

Rule-engine gọi billing mỗi khi tenant tạo rule. Alert quota cũng check đồng bộ trước khi evaluate.

| Tiêu chí | Đánh giá |
|----------|----------|
| Rule create latency | +10–15ms (1 REST call trong Docker network) — chấp nhận được cho CRUD |
| Alert hot path | Check đồng bộ trước mỗi TradeEvent → tăng latency evaluation đáng kể |
| Consistency | Strong cho cả hai |
| Complexity | Alert-delivery và billing phải sync trước khi eval — tạo tight coupling trên hot path |

**Hạn chế chính:** TradeEvent đến 80/giây — check quota đồng bộ trước mỗi eval = bottleneck trực tiếp trên hot path.

---

### Phương án C — Hybrid: Rule quota sync + Alert quota event-driven ✓

- **Rule quota** (P0, cold path): rule-engine gọi `billing GET /internal/quota/{tenantId}` khi CREATE rule; cache Caffeine 30s; nếu `rulesQuotaExceeded=true` → trả 402.
- **Alert quota** (P0, hot path): billing consume `AlertTriggeredEvent` → đếm usage → khi vượt → phát `QuotaExceededEvent` → rule-engine nhận → đánh dấu tenant `EVAL_PAUSED` → ngừng evaluate rule của tenant đó.
- **Gateway** không check quota — chỉ auth.

| Tiêu chí | Đánh giá |
|----------|----------|
| Rule create latency | +10ms (1 REST call) — OK cho CRUD |
| Alert hot path latency | Không bị chặn — hot path không chờ billing |
| Consistency | Rule: strong (30s cache); Alert: eventual ~1s |
| Coupling | Gateway stateless; rule-engine biết về quota state qua event — loose coupling |
| Complexity | Rule-engine cần in-memory quota state + warm lại khi restart |

---

### So sánh tổng hợp

| Tiêu chí | A: Gateway check | B: Rule-engine sync | C: Hybrid ✓ |
|----------|-----------------|---------------------|-------------|
| Gateway stateless | ✗ | ✓ | ✓ |
| Hot path không bị block | ✗ | ✗ | ✓ |
| Rule quota consistency | Strong | Strong | Strong (30s) |
| Alert quota consistency | Strong | Strong | Eventual (~1s) |
| Độ phức tạp | Trung bình | Cao | Trung bình |

---

## Quyết định

**Chọn Phương án C — Hybrid.**

Lý do:

1. **Gateway giữ stateless** — chỉ auth (SHA-256 + cache) + routing. Không phải biết về quota.

2. **Rule quota là cold path** — người dùng tạo rule không thường xuyên; +10ms cho 1 REST call chấp nhận được; `rulesQuotaExceeded` không thay đổi liên tục.

3. **Alert quota là hot path** — 80 trades/giây, mỗi trade đi qua rule evaluation; blocking sync check = bottleneck. Event-driven cho phép hot path chạy không bị chặn; khi billing phát `QuotaExceededEvent`, rule-engine nhận và pause — eventual consistency ~1s đủ cho usecase này.

---

## Chi tiết implementation

### Rule quota (sync)

```
POST /api/v1/rules
  → rule-engine: check in-memory quota cache
  → cache miss (hoặc hết 30s TTL): GET billing /internal/quota/{tenantId}
  → rulesQuotaExceeded = true → HTTP 402 { "error": "QUOTA_EXCEEDED_RULES" }
  → false → tạo rule → cache invalidate
```

Caffeine cache key: `tenantId`; TTL: 30s; evict khi rule CRUD thay đổi count.

### Alert quota (event-driven)

```
AlertTriggeredEvent → billing (consume)
  → INSERT/UPDATE usage_periods.alerts_fired++
  → IF alerts_fired > plan.max_alerts_pm:
      publish QuotaExceededEvent { tenantId, quotaType="ALERTS", ... }

QuotaExceededEvent → rule-engine (consume)
  → SET tenant.alertsQuotaExceeded = true (in-memory map)
  → evaluate(): nếu tenant bị flag → skip, không publish AlertTriggeredEvent
```

Rule-engine warm lại quota state khi restart: gọi `billing GET /internal/quota/{tenantId}` cho tất cả tenant có rule ACTIVE.

---

## Consequences

**Tích cực:**
- Hot path (trade → eval → alert) không bị blocking quota check.
- Gateway stateless — dễ scale horizontal nếu cần.
- Phân tách rõ: rule-engine biết về quota state, gateway không biết gì về business quota.

**Đánh đổi / hạn chế:**
- **Eventual consistency ~1s** cho alert quota: trong khoảnh khắc vượt quota, có thể thêm vài alert được bắn trước khi `QuotaExceededEvent` được xử lý. Chấp nhận được — không có tình huống nào trong Lab yêu cầu hard real-time quota enforcement trên hot path.
- **In-memory quota state** trong rule-engine cần warm lại khi restart — thêm startup dependency với `billing`.
- Nếu billing service down: rule-engine dùng cached state (stale tối đa 30s cho rule quota); alert quota enforcement bị delay cho đến khi billing back online.

/**
 * Auth filter chạy trước mọi route filter.
 * Flow: X-API-Key → SHA-256 → Caffeine cache(TTL 5s) → identity lookup → forward với X-Tenant-Id.
 * Loại trừ /actuator/** để compose healthcheck hoạt động.
 */
@Component
public class ApiKeyAuthFilter implements GlobalFilter, Ordered {

    private static final String HEADER_API_KEY = "X-API-Key";
    private static final String HEADER_TENANT_ID = "X-Tenant-Id";

    private final WebClient webClient;
    // Optional.of(tenantId) = active key; Optional.empty() = revoked/not-found (negative cache)
    private final Cache<String, Optional<String>> keyCache;

    public ApiKeyAuthFilter(@Value("${identity.service.url}") String identityServiceUrl) {
        this.webClient = WebClient.builder().baseUrl(identityServiceUrl).build();
        this.keyCache = Caffeine.newBuilder()
                .expireAfterWrite(5, TimeUnit.SECONDS)
                .maximumSize(10_000)
                .build();
    }

    @Override
    public int getOrder() {
        return Ordered.HIGHEST_PRECEDENCE;
    }

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String path = exchange.getRequest().getURI().getPath();
        if (path.startsWith("/actuator")) {
            return chain.filter(exchange);
        }

        String rawKey = exchange.getRequest().getHeaders().getFirst(HEADER_API_KEY);
        if (rawKey == null || rawKey.isBlank()) {
            return reject(exchange, HttpStatus.UNAUTHORIZED, "MISSING_API_KEY");
        }

        String keyHash = sha256Hex(rawKey);

        Optional<String> cached = keyCache.getIfPresent(keyHash);
        if (cached != null) {
            if (cached.isEmpty()) {
                return reject(exchange, HttpStatus.UNAUTHORIZED, "INVALID_API_KEY");
            }
            return forward(exchange, chain, cached.get());
        }

        return webClient.get()
                .uri("/internal/keys/" + keyHash)
                .retrieve()
                .bodyToMono(KeyLookupResponse.class)
                .timeout(Duration.ofSeconds(2))
                .flatMap(resp -> {
                    keyCache.put(keyHash, Optional.of(resp.tenantId()));
                    return forward(exchange, chain, resp.tenantId());
                })
                .onErrorResume(e -> {
                    if (e instanceof WebClientResponseException wex && wex.getStatusCode().value() == 404) {
                        keyCache.put(keyHash, Optional.empty());
                        return reject(exchange, HttpStatus.UNAUTHORIZED, "INVALID_API_KEY");
                    }
                    // 5xx hoặc timeout — không cache để gateway thử lại khi identity hồi phục
                    return reject(exchange, HttpStatus.SERVICE_UNAVAILABLE, "IDENTITY_UNAVAILABLE");
                });
    }

    private Mono<Void> forward(ServerWebExchange exchange, GatewayFilterChain chain, String tenantId) {
        ServerHttpRequest mutated = exchange.getRequest().mutate()
                .header(HEADER_TENANT_ID, tenantId)
                .headers(h -> h.remove(HEADER_API_KEY))
                .build();
        return chain.filter(exchange.mutate().request(mutated).build());
    }

    private Mono<Void> reject(ServerWebExchange exchange, HttpStatus status, String errorCode) {
        exchange.getResponse().setStatusCode(status);
        exchange.getResponse().getHeaders().setContentType(MediaType.APPLICATION_JSON);
        byte[] body = ("{\"error\":\"" + errorCode + "\"}").getBytes(StandardCharsets.UTF_8);
        return exchange.getResponse().writeWith(
                Mono.just(exchange.getResponse().bufferFactory().wrap(body)));
    }

    private static String sha256Hex(String input) {
        try {
            MessageDigest md = MessageDigest.getInstance("SHA-256");
            byte[] hash = md.digest(input.getBytes(StandardCharsets.UTF_8));
            StringBuilder sb = new StringBuilder(64);
            for (byte b : hash) {
                sb.append(String.format("%02x", b));
            }
            return sb.toString();
        } catch (NoSuchAlgorithmException e) {
            throw new IllegalStateException("SHA-256 unavailable", e);
        }
    }

    private record KeyLookupResponse(String tenantId, String planId, String status) {}
}

import jakarta.validation.Valid;
import org.springframework.http.HttpStatus;
import org.springframework.web.bind.annotation.*;

import java.util.List;
import java.util.UUID;

@RestController
public class KeyController {

    private final ApiKeyService apiKeyService;

    public KeyController(ApiKeyService apiKeyService) {
        this.apiKeyService = apiKeyService;
    }

    @PostMapping("/api/v1/keys")
    public CreateKeyResponse createKey(
            @RequestHeader("X-Tenant-Id") UUID tenantId,
            @Valid @RequestBody CreateKeyRequest request
    ) {
        return apiKeyService.issue(tenantId, request.label());
    }

    @GetMapping("/api/v1/keys")
    public List<KeySummaryResponse> listKeys(
            @RequestHeader("X-Tenant-Id") UUID tenantId
    ) {
        return apiKeyService.listByTenant(tenantId);
    }

    @DeleteMapping("/api/v1/keys/{id}")
    @ResponseStatus(HttpStatus.NO_CONTENT)
    public void deleteKey(
            @RequestHeader("X-Tenant-Id") UUID tenantId,
            @PathVariable UUID id
    ) {
        apiKeyService.revoke(tenantId, id);
    }
}
